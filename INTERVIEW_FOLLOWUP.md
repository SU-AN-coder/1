# 面试后补充说明：多 Agent 协作平台 MVP

## 说明

您好，

在上一轮面试中，我们聊到了多 Agent 协作、GitHub 工作流编排、契约门禁、任务流转、冲突处理，以及如何把 GitHub Enterprise 作为协作底座来设计一套可治理的研发平台。当时我对其中一些问题的回答不够完整，面试后我把这些问题重新系统梳理了一遍，并基于这些讨论做了一个可运行的 MVP 原型。

这个项目不是对面试问题的“事后背答案”，而是我希望用一个更具体的方式，展示自己对这些问题的复盘、理解和工程化落地能力。

## 项目目标

项目名称：`Multi-Agent Collaboration Platform MVP`

目标是构建一层轻量的编排系统，复用 GitHub Enterprise 已有的仓库、分支、PR、Review、Check-run 和权限体系，在其上增加：

- 任务状态机
- 契约驱动的开发门禁
- 契约变更后的级联阻塞与恢复
- 有限协商与人工升级机制
- GitHub webhook 驱动的 PR/冲突状态处理
- 审计与事件记录

我把当时讨论的核心设计思路整理在技术文档里，并做了对应的代码原型。

## 当前实现到什么程度

截至 2026-03-13，这个仓库已经实现了一个可运行的内存版 MVP，包含：

- 基于任务状态机的任务流转
- 开发任务依赖契约合并后的解锁流程
- 契约修改进入 review 后，对下游任务进行级联阻塞
- 契约再次合并后，要求下游任务先同步再恢复
- P2P 协商最多 3 回合，超限后升级为人工审批
- GitHub webhook 驱动的 merge conflict 检测与人工兜底
- 事件日志与审计日志
- 仓储接口抽象，为后续接入 PostgreSQL / Redis 做准备

## 当前项目状态

这个项目目前适合被定义为：

`一个可运行、可测试、可继续扩展的 MVP 编排层原型`

而不是：

`已经完成生产级基础设施接入的平台`

目前已经验证通过：

- `npm run build`
- `npm test`

当前测试通过情况：

- 6 个测试全部通过

## 已经完成的部分

代码中已经落地的重点包括：

- 平台编排核心逻辑
- Fastify API 层
- 本地最小 Repo Context Tool
- 内存版仓储实现与后续持久化接入抽象

相关入口文件包括：

- `AA.md`

## 还没有完成的部分

为了保持 MVP 的边界清晰，目前我还没有把以下内容接成生产可用版本：

- PostgreSQL 持久化
- Redis 幂等键 / 锁 / 队列
- GitHub App 真实鉴权与 webhook 签名校验
- 生产环境部署配置
- 更严格的权限模型与组织级策略校验

这部分我也已经补了运维准备清单，目的是把“从原型走向真实接入”需要准备的外部资源先理顺。

## 我从这次复盘里补强了什么

这次补做项目，对我来说最重要的不是“把代码写出来”，而是把几个原来回答得不够扎实的问题真正落到工程层面：

- 如何把“契约先行”变成状态机和门禁，而不是停留在口头原则
- 如何处理契约修改对下游任务的影响
- 如何限制 Agent 间协商，避免无限来回
- 如何把 GitHub 作为协作事实源，而不是额外造一个聊天式系统
- 如何给后续数据库、缓存、GitHub App 接入留出清晰边界

我也因此更清楚地区分了：

- 哪些是“架构原则”
- 哪些是“最小可运行 MVP”
- 哪些是“真实生产接入前必须补齐的基础设施”
