## 说明

Andy,你好

前天的面试中，我们聊到了一个团队下，多个 Agent 协作、GitHub 工作流编排，还有Agent之间如何进行协作 传输消息，我会采用怎么样的架构来完成Agent的协作问题。这些本质上是对于之前task测试题在真正的团队项目如何去应用，来展开进一步的探索。当时的我对其中的问题的回答不够完善，面试后我把这些问题重新梳理了一遍，并对这些讨论做了一个可运行的 MVP 。

## 项目

目标是构建一层轻量的编排系统，复用 GitHub Enterprise 已有的仓库、分支、PR、Review、Check-run 和权限体系，在其上增加：

- 任务状态机
- 契约驱动的开发门禁
- 契约变更后的级联阻塞与恢复
- 有限协商与人工升级机制
- GitHub webhook 驱动的 PR/冲突状态处理
- 审计与事件记录

我把当时讨论的思路整理在技术文档[AA.md](AA.md)里，并做了对应的代码。

## 当前已实现

这个仓库已经实现了一个可运行的内存版 MVP，包含：

- 基于任务状态机的任务流转
- 开发任务依赖契约合并后的解锁流程
- 契约修改进入 review 后，对下游任务进行级联阻塞
- 契约再次合并后，要求下游任务先同步再恢复
- P2P Agent协商最多 3 回合，超限后升级为人工审批
- GitHub webhook 驱动的 merge conflict 检测与人工兜底
- 事件日志与审计日志
- 仓储接口抽象，为后续接入 PostgreSQL / Redis 做准备

这个项目目前适合被定义为：

`一个可运行、可测试、可继续扩展的 MVP 编排层原型。但未完成生产级基础设施接入`


代码中已经落地的：

- 任务与契约相关领域模型定义
- 平台编排核心逻辑
- Fastify API 层
- 本地最小 Repo Context Tool
- 内存版仓储实现与后续持久化接入抽象
